How is the result decided?

((SeedA + SeedB + SeedC) / (2 ^ 256 / 100) ) +1 = result

SeedA = Is set before you place your bet, and we cannot change it. Basically; when the first bet of the round is placed, the contract will retrieve and store the blockhash of the parent block relative to the block the first bet for the round was made on. The contract uses that as its SeedA number. See 'provably fair' below for more details. 1st dice rolled.

SeedB = Your number choice between 1-100, when you are the first better for that round. If a game is already in progress, and you are submitting late bets to the round, SeedB is generated by the first player to submit a bet in that round. 2nd dice rolled.

SeedC = Hash of block in a future time. 3rd dice rolled.

Before any bets are placed, we have already generated a number from a previous block hash. This is SeedA.

Once a player submits a bet along with a number (SeedB) for a particular round, the Etheroll contract waits for a set number of new Ethereum blocks to be mined, so that we can the use a future block hash from which SeedC generated.

Therefore SeedC is impossible for us, or you to know, prior to the submission of the bet as the block did not exist on the Blockchain at the time of your bet.

Provably Fair

SeedA is generated by a block hash previous to your bet. By the time you make your bet, SeedA was already generated by the Ethereum Blockchain, we pull the block hash information from that block to use as SeedA.

You can check the block number and compare its block hash to sha3 using any good block explorer. This will confirm that we did indeed use the number we said we did (a number that already existed before you made your bet). We provide the sha3 encrypted version of SeedA ('SHA3 of SeedA'). This is to prove that we do not (and could not - mathematical impossibility) change SeedA after you submit your bet. This means Etheroll will always have picked a number before you pick yours.
